```plain
在渗透测试中，在获取目标外网权限后，需要通过转发端口或搭建代理等方式建立内网通道。本节课将简要介绍这些转发和代理技术的相关基础，以及搭建相应测试环境，通过演示常规工具的使用让各位同学更好理解。
```

## 一、端口转发和代理
### 1.1、正向连接和反向连接
```plain
在开始介绍端口转发与内网代理前，先补充两个基本概念：正向连接和反向连接。例如，Metasploit 大致可以分为两种 Meterpreter，一种是以 windows/meterpreter/bind_tcp为代表的Bind Shell，另一种是以 windows/meterpreter/reverse_tcp 为代表的 Reverse Shell。其中，Bind Shell 用于正向连接，而Reverse Shell 用于反向连接。
```

#### 1.1.1、正向连接
```plain
正向连接就是受控端主机监听一个端口，由控制端主机主动去连接受控端主机的过程，适用于受控主机具有公网IP的情况下。例如在下图中，攻击者 和 受害者 主机都具有公网IP，攻击者 可以直接通过IP地址访问到 受害者，所以能够使用正向连接来控制 受害者。
```

#### 1.1.2、反向连接
```plain
反向连接是控制端主机监听一个端口，由受控端主机反向去连接控制端主机的过程，适用于受控端主机没有公网IP的情况。例如，如下图所示，受害者 是一台位于内网，并且没有公网IP的主机，攻击者无法直接通过IP地址访问到 受害者。所以此时需要在Atacker 上监听一个端口，让受害者去反向连接 攻击者，从而实现对 受害者的控制。在渗透测试中，正向连接往往受限于受控主机上的防火墙屏蔽及权限不足等情况，而反向连接可以很好地突破这些限制。
```

### 1.2、端口转发
```plain
端口转发（Port Forwarding）是网络地址转换（NAT）的一种应用。通过端口转发，一个网络端口上收到的数据可以被转发给另一个网络端口。转发的端口可以是本机的端口，也可以是其他主机上的端口。

在现实环境中，内网部署的各种防火墙和入侵检测设备会检查敏感端口上的连接情况，如果发现连接存在异样，就会立即阻断通信。通过端口转发，设置将这个被检测的敏感端口的数据转发到防火墙允许的端口上，建立起一个通信隧道，可以绕过防火墙的检测，并与指定端口进行通信。

端口映射（Port Mapping）也是网络地址转换（NAT）的一种应用，用于把公网的地址翻译成私有地址。端口映射可以将外网主机收到的请求映射到内网主机上，使得没有公网 IP 地址的内网主机能够对外提供相应的服务。
```

注意，根据相关资料，端口转发与端口映射的概念并没有严格的术语解释，有的资料只是定义了这两个术语，并作为同一个术语进行解释，所以我们此处也不做区分。

### 1.3、SOCKS代理
```plain
SOCKS 全称为 Protocol For Sessions Traversal Across Firewall Securely，是一种代理协议，其标准端口为1080。SOCKS 代理有 SOCKS 4 和 SOCKS 5 两个版本，SOCKS 4  只支持TCP，而 SOCKS 5 在 SOCKS 4 的基础上进一步扩展，可以支持 UDP 和各种身份验证机制等协议。采用 SOCKS 协议的代理服务器被称为 SOCKS 服务器，这是一种通用的代理服务器，在网络通信中扮演着一个请求代理人的角色。在内网渗透中，通过搭建 SOCKS 代理，可以与目标内网主机进行通信，避免多次使用端口转发。
```

```plain
lcx是一个基于 Soket 套接字实现的端口转发工具，有 Windows 和 Linux 两个版本。一个正常的 Socket 隧道必须具备两端：一个为服务端，监听一个端口，等待客户端的连接；另一端为客户端，通过传入服务端的 IP 地址和端口，才能主动与服务器连接。
```

```plain
假设此时已经获取了 Windows Server 2012 的控制权，需要登录其远程桌面查看情况，但是防火墙对 3389 端口做了限制，不允许外网机器对 3389 端口进行连接。那么，通过端口转发，可以将3389端口转发到其他防火墙允许的端口上，如4444端口。

在  Windows Server 2012 上执行以下命令，然后通过连接 Windows Server 2012 的 4444 端口，即可成功访问其远程桌面。
```

```plain
lcx.exe -tran 4444 127.0.0.1:3389

- lcx.exe 是local port forwarding工具的执行程序。
- -tran 参数表示启动端口转发。
- 4444 是本地端口,用于监听连接。
- 127.0.0.1:3389 是远程主机地址和端口。

所以这个命令的作用是:

1. 在本地监听 4444 端口。
2. 当有连接到本地4444端口时,lcx会自动建立与127.0.0.1地址的3389端口的连接。
3. 随后转发本地4444端口和远程3389端口之间的数据流。

这样就实现了本地端口4444到远程端口3389的转发。本地程序连接4444,就相当于连接了远程3389端口
```

### 2.4、Netsh实现端口转发
**Netsh** 是Windows自带的命令行脚本工具，它可以建立端口映射。

```plain
现在有这么一个环境，内网中有一台Web服务器，但是我们处于公网，所以无法访问该服务器。于是，我们可以在中间Web服务器上利用Netsh实现一个端口映射，只要我们访问中间Web服务器公网地址的指定端口，就相当于我们访问内网Web服务器的80端口。
```

```plain
netsh interface portproxy add v4tov4 listenaddress=WEB服务器 listenport=8080 connectaddress=内网主机 connectport=80 #新建一个端口映射，将WEB服务器的8080端口和内网主机的80端口做个映射

命令解析:
netsh interface portproxy add : 在netsh接口下添加端口转发规则
v4tov4: 指定转发的为IPV4到IPV4流量
listenaddress=WEB服务器: 设置外网可以访问的WEB服务器地址
listenport=8080: 设置在WEB服务器上监听的端口为8080
connectaddress=内网主机: 设置要转发到的内网主机地址
connectport=80: 设置要转发到内网主机的端口80

netsh interface portproxy show all  #查看端口映射

netsh interface portproxy delete v4tov4 listenaddress=WEB服务器 listenport=8080 #删除端口映射
```

## 三、利用ICMP进行隧道穿透

## 四、利用SSH进行隧道穿透
在实际环境中，绝大部分Linux/Unix服务器和网络设备支持SSH协议，运维人员通常会使用SSH命令来连接远程主机，内网中的主机设备及边界防火墙一般都会允许SSH协议通过。但是SSH的功能不止于此，它还可以用来进行流量转发，转发TCP端口的数据流量，并且在流量传输过程中数据是加密的，这意味着可以利用SSH搭建隧道突破防火墙的限制。当其他协议方式无法利用时，可以尝试使用SSH进行穿透。

实验环境如下

### 4.1 本地端口转发
需要满足的条件是确保目标服务器的22端口开放，同时还需要获取到对方的SSH账号和密码。满足上述条件后，先通过攻击机连接Web服务器。因Web服务器可以与靶机通过192.168.72.0/24这个网段进行通信，通过-L参数指定靶机3389端口转发到攻击机的8888端口，在攻击机中执行下方命令，即可成功执行本地端口转发。

```plain
ssh -L *:8888:靶机IP:3389 root@WEB服务器IP

#表示从任意主机发往攻击机 8888 端口的流量，都通过 WEB 服务器，发往靶机的 3389 端口。
```

### 4.3 动态端口转发
SSH动态端口转发就是建立起一个 SSH 加密的 SOCKS 4/5 代理隧道，任何支持 SOCKS4/5 协议的程序都可以通过它进行代理访问。在攻击机中执行下方命令建立 SOCKS 代理通道，这里需要输入 Web 服务器的密码。输入密码后，执行成功。

```plain
ssh -D *:8888 root@Web服务器
```

```plain
socks5 127.0.0.1 8888
```

#  SOCKS4、SOCKS4a 与 SOCKS5 对比  
| **<font style="color:rgb(31, 31, 31);">特性</font>** | **<font style="color:rgb(31, 31, 31);">SOCKS4</font>** | **<font style="color:rgb(31, 31, 31);">SOCKS4a</font>** | **<font style="color:rgb(31, 31, 31);">SOCKS5</font>** |
| --- | --- | --- | --- |
| **<font style="color:rgb(31, 31, 31);">身份认证</font>** | <font style="color:rgb(31, 31, 31);">不支持</font> | <font style="color:rgb(31, 31, 31);">不支持</font> | **<font style="color:rgb(31, 31, 31);">支持</font>**<font style="color:rgb(31, 31, 31);">（多种方式，如用户名/密码）</font> |
| **<font style="color:rgb(31, 31, 31);">传输协议</font>** | <font style="color:rgb(31, 31, 31);">仅限 TCP</font> | <font style="color:rgb(31, 31, 31);">仅限 TCP</font> | **<font style="color:rgb(31, 31, 31);">TCP 和 UDP</font>** |
| **<font style="color:rgb(31, 31, 31);">域名解析</font>** | **<font style="color:rgb(31, 31, 31);">本地解析</font>**<font style="color:rgb(31, 31, 31);">（客户端解析）</font> | **<font style="color:rgb(31, 31, 31);">远程解析</font>**<font style="color:rgb(31, 31, 31);">（代理服务器解析）</font> | **<font style="color:rgb(31, 31, 31);">远程解析</font>**<font style="color:rgb(31, 31, 31);">（支持更完美）</font> |

| **<font style="color:rgb(31, 31, 31);">应用场景</font>** | <font style="color:rgb(31, 31, 31);">极少见（老旧设备）</font> | <font style="color:rgb(31, 31, 31);">较少见</font> | **<font style="color:rgb(31, 31, 31);">主流标准</font>** |
